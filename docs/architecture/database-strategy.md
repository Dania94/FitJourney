# Purpose

This document defines the database principles and conventions used by FitJourney.

The goal is to create a database that is:

- maintainable,
- scalable,
- modular,
- future-proof,
- ready for future feature growth.

---

# Database Technology

## Relational Database

PostgreSQL

### Rationale

- Mature and reliable.
- Excellent support in Spring Boot.
- Strong SQL capabilities.
- Supports advanced features such as JSON and full-text search.
- Widely adopted in enterprise environments.

---

# Database Architecture

## Single Database

The application uses a single PostgreSQL database.

---

## Multiple Schemas

The database is divided into schemas according to module ownership.

### Schemas

- user
- nutrition
- workout
- progress
- recommendation
- womens_health
- notification

### Example

user.users

nutrition.foods

nutrition.food_logs

workout.workout_plans

---

# Benefits

- Clear ownership.
- Better modular boundaries.
- Easier future extraction to microservices.
- Better organization.

---

# Primary Keys

## UUID

All entities use UUID as their primary key.

### Example

```sql
id UUID PRIMARY KEY
```

### Rationale

- Globally unique identifiers.
- Easier migration to distributed systems.
- IDs can be generated by the application.
- Better security than sequential IDs.

---

# Auditing

Most tables should support auditing.

## Standard Audit Columns

```sql
created_at TIMESTAMP WITH TIME ZONE
created_by UUID
updated_at TIMESTAMP WITH TIME ZONE
updated_by UUID
```

### Purpose

- Track changes.
- Debugging.
- Historical information.
- Operational support.

---

# Soft Delete

Soft delete will be used selectively.

Instead of:

```sql
DELETE FROM users;
```

we mark the record as deleted.

```sql
deleted_at TIMESTAMP WITH TIME ZONE
deleted_by UUID
```

---

# Entities Using Soft Delete

- users
- foods
- exercises
- meal_plans
- workout_plans

---

# Entities Not Using Soft Delete

- weight_entries
- measurements
- notifications
- food_logs
- workout_sessions

---

# Time Handling

All timestamps are stored in UTC.

## Java

```java
Instant
```

## PostgreSQL

```sql
TIMESTAMP WITH TIME ZONE
```

# Naming Conventions

## Columns

# Foreign Keys

Every relationship should explicitly define foreign keys.

Example:

```sql
user_id UUID NOT NULL
```

---

# Future Sharing Requirements

The application will support sharing:

- meal plans
- workout plans

Permissions:

- OWNER
- EDITOR
- VIEWER

External users can access shared content through read-only links.

# Flyway

All database changes must be managed using Flyway migrations.

Example:

V1__create_user_schema.sql

V2__create_users_table.sql

V3__create_nutrition_schema.sql

---

# Database Principles

1. PostgreSQL database.
2. Multiple schemas.
3. UUID primary keys.
4. Auditing support.
5. Selective soft delete.
6. UTC timestamps.
7. Flyway migrations.
8. Explicit foreign keys.
9. Future sharing support.

---

# Decision Summary

FitJourney uses a single PostgreSQL database with multiple schemas, UUID identifiers, auditing support, and selective soft delete to provide a maintainable and scalable database architecture.
